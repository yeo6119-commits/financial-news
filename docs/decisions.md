# 설계 결정과 이유

각 결정은 실제 문제를 진단한 결과다. **되돌리기 전에 왜 그랬는지 먼저 읽을 것.**

---

## 건드리면 안 되는 것

### `batch: false` — 요약 배치 금지

기사 여러 건을 한 프롬프트에 묶으면 토큰은 아끼지만 **모델이 일부 기사를 조용히 누락시킨다.** 초기 기사 유실의 원인으로 진단됐다. 기사 1건 = 호출 1회를 지킨다.

### VACUUM은 트랜잭션 밖에서

```python
conn.commit()
conn.isolation_level = None
conn.execute("VACUUM")
conn.isolation_level = ""
```

이 수정이 **5번 반복해서 날아갔다.** `db.py`를 새로 만들 때마다 빠뜨렸기 때문. 워크플로 초록불의 전제 조건이다.

### `sports_exclude`에 일반어 금지

"경기", "시즌", "리그", "우승", "전반기/후반기", "선수", "감독"을 넣으면 안 된다.

- **경기** → 경기도 지역 기사, 경기 전망 기사
- **시즌** → 시즌 프로모션
- **리그** → 리그오브레전드
- **우승** → 공모전 우승
- **전반기/후반기** → 반기 실적

실제로 "경기"를 넣었다가 경기도 기사가 전멸한 사고가 있었다. KBO·탈삼진·시청률처럼 **스포츠에만 쓰이는 말만** 넣는다.

### `KB` 단독 키워드 금지

`group_keywords`에 `KB`를 넣으면 **`KBO`가 매칭된다.** 야구 기사가 통과했던 원인. 또 `KB`가 브랜드 목록에 들어가면 관련성 키워드로 자동 편입되어 **모든 KB 기사가 무조건 통과**한다. 정식 명칭(`KB금융`, `KB국민은행`)만 쓴다.

## 수집

### 검색 기간 = 요청 시각 기준 24시간 고정

**대안이었던 것**: 직전 실행 시각 기준

**이유**: 매일 안 돌릴 수도 있어서 일관성이 없어진다. 월요일 예외 처리, 오버랩, 72시간 상한 같은 복잡도가 붙는다.

**감수사항**: 하루 이상 건너뛰면 그 사이 기사는 못 잡는다. 대신 자주 돌려도 "기열람" 중복 제거로 같은 기사가 반복 노출되지 않는다.

### 네이버·카카오 본사 제외

금융회사가 아니다. 게다가 인터넷·핀테크 그룹은 `standalone_mode: true`라 회사명이 그대로 검색어가 되는데, "네이버" 단독 검색은 웹툰·쇼핑·스포츠를 전부 긁어온다. 실제로 이것 때문에 실행이 13분으로 늘고 "절단: 네이버" 경고가 떴다.

`네이버페이`, `네이버파이낸셜`, `카카오뱅크`, `카카오페이` 등 **금융 계열만** 유지.

## 필터링

### 수집은 넓게, 판단은 본문 확보 후 로컬에서

검색 단계에서 미리 좁히면 놓치는 기사가 생긴다. 넓게 긁어온 뒤 로컬에서 판단한다. 네이버 API는 무료이므로 수집량은 비용이 아니다.

### 1차 스크리닝은 제목만으로

본문 추출은 건당 1~5초다. 1,800건을 다 긁으면 실행이 20분을 넘는다. 제목만으로 걸러 **~300건**만 추출한다.

### 제목에 단서 없으면 본문 리드로 판정

"KB국민은행, 새 시스템 가동" 같은 제목은 디지털 기사인지 알 수 없다. 이런 건 본문 **리드 400자**를 읽어 판정한다.

**리드만 보는 이유**: 한국 기사는 역피라미드라 주제가 리드에 온다. 뒤쪽 언급은 그 기사의 주제가 아니다.

**핵심 키워드만 인정하는 이유**: 관련성 목록 전체(122개)엔 앱 이름이 포함되어, CSR 기사에 "신청은 하나원큐 앱에서" 한 줄만 있어도 통과했다. 삼계탕 나눔 기사가 이렇게 뚫렸다. 주제어급 **49개**만 인정한다.

### 본문 읽을 가치가 없으면 미리 제외

`NON_DIGITAL_TOPICS` 63개(CSR·금리·실적·행사). 읽어봐야 어차피 제외될 기사에 1~5초를 쓸 이유가 없다. 이 규칙 없이는 실행이 **21분**까지 늘었다.

**순서가 중요하다** — 관련성 검사를 먼저 하므로 "AI 기반 금리 산정"은 '금리'가 있어도 통과한다.

### 종합기사는 태그 이름이 아니라 구조로 판정

`[주목! e금융]`, `[여의도단신]`, `[금융레이더]` — 태그 이름은 매체마다 계속 새로 생겨서 열거 방식은 한계가 있다. 또 회사명이 `하나`, `KB국민`처럼 잘려 매칭이 안 되기도 한다.

**대괄호 태그 + 구분자(·, /) 3개 이상** → 종합기사. 구조로 잡으면 새 태그도 자동으로 걸린다.

### 브랜드명은 관련성 키워드

`SOL LINK`, `하나원큐`, `KB Pay` 같은 앱·서비스 이름이 제목에 있으면 **그 자체로 디지털 기사**다. `BRAND_TO_COMPANY`의 키를 관련성 목록에 자동 편입한다. 매핑을 한 곳(classifier)에서만 관리하므로 새 앱은 한 번만 추가하면 된다.

### 보안 키워드는 디지털 맥락이 분명한 것만

`금융사기 예방`, `인증 강화`는 뺐다. 오프라인 노인 대상 예방교육 기사가 통과하던 원인. `보이스피싱`·`FDS`·`딥페이크`처럼 디지털 사안임이 분명한 것만 남긴다.

## 중복 제거

### 제목 토큰 방식이 주력

한국 기사는 **본문 simhash만으로 안 잡힌다.** 같은 사건을 다룬 기사끼리도 해밍거리가 14~28로 벌어진다. 제목 토큰 겹침이 훨씬 정확하다.

**실측**: Fast-Fit 송금 8건 → 3건, 탭앤페이 4건 → 1건, 카카오뱅크 몽골 6건 → 1건

### 한국어 조사 제거

`공무원연금공단`과 `공무원연금공단과`가 다른 토큰으로 갈려 중복을 놓쳤다.

**실측**: 신한은행 공무원연금공단 2건 겹침 0.20 → **0.40** (묶임), 다른 사건은 0.20 유지 (안 묶임)

어간이 2자 이상 남을 때만 조사를 뗀다.

## 요약

### 줄마다 역할 지정하지 않음

예전엔 "1줄 도입 / 2줄 변화 / 3줄 시사점"이었다. 본문 사실만 고르게 담는 방식으로 바꾼 이유:

1. 실제 용도가 **사실 파악**이지 임원 보고가 아님
2. 프롬프트가 짧아짐 (1,312자 → 430자)
3. **8b 모델도 충분히 해냄** — 70b와의 격차가 주로 "3번째 줄 추론"에서 났는데 그 요구가 사라짐

### 근거 검증 (`_grounded`)

모델이 요약할 내용을 못 찾으면 **프롬프트의 예시를 그대로 복창한다.** 골드만삭스 기사와 JP모건 기사에 똑같은 "KB AI Lab" 요약이 나온 실제 사고.

요약의 명사 토큰이 원문에 55% 이상 등장하는지 검증한다. 예시 복창은 겹침 **0.00**, 정상 요약은 **0.81**로 명확히 갈린다. 예시 복창뿐 아니라 모든 환각을 잡는다.

### 형식 실패는 1회만 재시도

형식 오류는 같은 프롬프트로 다시 불러도 대개 같은 결과다. 3회 재시도 + 8b 3회 = 최대 6회는 **한 건에 10,038토큰**(정상 6건치)을 태운다. 실패 8건이면 하루 쿼터의 절반이 증발한다.

### 본문 200자 미만은 호출 생략

토큰을 아끼면서 동시에 **환각의 근본 원인**을 없앤다. 짧은 본문을 받은 모델이 예시를 복창하는 것이 KB AI Lab 사고의 메커니즘이었다.

### 본문 정제 후 문장 단위 절단

실측 결과 본문의 **37%가 군더더기**(기자 프리픽스, 이메일, 사진 캡션, 저작권 문구)였다. 정제 후 자르면 같은 토큰에 실질 내용이 훨씬 많이 담긴다.

문장 중간에서 자르면 모델이 끊긴 문장을 해석하느라 품질도 떨어지고 토큰도 낭비된다.

## 저장

### 제외 기사는 DB에 저장하지 않음

HTML의 제외 목록 패널은 **현재 회차 것만** 보여준다. 즉 과거 회차 제외분은 저장만 되고 아무도 안 봤다. 메모리에서 HTML로 직접 전달하면 검수 기능은 그대로면서 DB만 가벼워진다.

**결과**: 6.4MB → 233KB (96% 감소)

중복 제거와 요약 캐시는 `delivered=1` 기사만 참조하므로 영향 없다.

### 본문 전문 미저장

`store_full_body: false`. DB 크기 절감. 대신 `retry_summary.py`는 URL로 본문을 다시 긁는다.

### HTML 생성 성공 후 delivered 마킹

생성이 실패하면 다음 실행에서 같은 기사를 다시 시도한다. 기사가 조용히 사라지지 않게 하는 장치.
