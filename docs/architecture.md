# 아키텍처

`main.py`가 8단계를 순서대로 실행한다. 각 단계는 앞 단계의 출력을 받아 `article` 딕셔너리에 필드를 채워 나간다.

---

## 파이프라인 8단계

### 1. 수집 (`collector.py`, `press_collector.py`)

| 소스 | 방식 | 결과 |
|---|---|---|
| 네이버 뉴스 API | 985개 검색어 × 페이지네이션, 8스레드 | ~1,800건 |
| 보도자료 | 9개 금융그룹 사이트 크롤링 | 수십 건 |

- **검색 기간**: 요청 시각 기준 **최근 24시간 고정** (`time.window_hours: 24`)
- **검색어 생성**: 회사명 × 서비스 키워드 조합. 인터넷·핀테크 그룹만 `standalone_mode: true`라 회사명 단독 검색
- **정렬**: `sort=date`, 24시간보다 오래된 페이지가 나오면 중단
- 수집 직후 `url_hash`로 뉴스·PR 간 완전 중복 제거

> **네이버 본사, 카카오 본사는 의도적으로 제외.** 금융회사가 아니고, 단독 검색 시 웹툰·쇼핑 노이즈가 폭증한다. `네이버페이`·`카카오뱅크` 등 금융 계열만 등록.

### 2. 1차 스크리닝 (`filter.prescreen`)

**제목만으로** 판정한다. 본문 추출은 건당 1~5초가 들므로, 읽어봐야 소용없는 건 미리 걸러 시간을 아낀다.

```
~1,800건 → ~300건
```

보도자료는 스크리닝 면제. 상세 규칙은 [filtering.md](filtering.md).

### 3. 본문 추출 (`extractor.py`)

- 병렬 **16스레드**, 타임아웃 **5초**
- 네이버 뉴스는 전용 셀렉터(`#dic_area` 등), 그 외는 범용 셀렉터 → p 태그 밀도 순으로 시도
- 실패해도 기사는 버리지 않고 `extract_ok=0`으로 표시

### 4. 2차 필터 (`filter.apply_filters`)

본문을 확보한 뒤 최종 판정. 1차에서 `_needs_body_check=True`로 넘어온 기사(제목에 회사명은 있으나 디지털·AI 키워드가 없는 것)를 **본문 리드 400자**로 판정한다.

### 5. 중복 제거 (`deduplicator.py`)

**회차 내** — 같은 사건을 다룬 기사 여러 건을 하나로 묶는다.

| 조건 | 임계값 |
|---|---|
| 같은 날 + 제목 토큰 겹침 | ≥ 0.35 |
| 제목 토큰 겹침 (날짜 무관) | ≥ 0.5 |
| 제목 유사도 | ≥ 0.62 |
| 본문 simhash 해밍거리 | ≤ 6 |

**회차 간** — 이미 반영한 기사는 다시 안 보여준다 (최근 3일 `delivered=1` 대상).

| 조건 | 임계값 |
|---|---|
| 본문 simhash 해밍거리 | ≤ 3 |
| 제목 토큰 겹침 | ≥ 0.5 |

> 한국 기사는 본문 simhash만으로는 잘 안 잡힌다. 같은 사건 기사끼리도 해밍거리가 14~28로 벌어진다. 그래서 **제목 토큰 방식**을 주력으로 쓴다.
>
> 토큰화 시 **한국어 조사를 제거**한다. `공무원연금공단`과 `공무원연금공단과`가 다른 토큰으로 갈려 중복을 놓치던 문제를 해결.

### 6. 분류 (`classifier.py`)

검색에 쓴 `menu_id`를 신뢰하지 않고 **제목 기준으로 재판정**한다.

- **긴 이름 우선 매칭** — `신한투자증권`을 `신한`보다 먼저 매칭
- **앞에 나온 것 우선** — 제목에 여러 회사가 있으면 먼저 등장한 쪽
- **브랜드 → 회사 정규화** — `하나원큐`→하나은행, `SOL LINK`→신한투자증권, `모니모`→삼성금융네트웍스 등 40여 개 (`BRAND_TO_COMPANY`)

### 7. 요약 (`summarizer.py`)

- **캐시 우선** — 같은 URL을 이미 요약했으면 Groq 호출 0회
- **기사 1건 = 호출 1회** (배치 절대 금지)
- 상세 내용은 [summarization.md](summarization.md)

### 8. HTML 생성 + DB 저장 (`html_generator.py`, `db.py`)

HTML 생성이 성공해야만 `delivered=1` 마킹. 생성이 실패하면 다음 실행에서 같은 기사를 다시 시도한다.

## 데이터 흐름

`article` 딕셔너리가 단계를 거치며 필드를 채운다.

```python
# 1. 수집 직후
{title, press, pub_date, original_url, naver_url, url_hash,
 norm_title, norm_title_hash, search_keyword, menu_id}

# 2. 스크리닝 후
{..., excluded, exclude_reason, _needs_body_check}

# 3. 추출 후
{..., body, extract_ok, extract_fail_reason}

# 4-5. 필터·중복 후
{..., body_hash, body_fingerprint, _body_relevance}

# 6. 분류 후
{..., fin_group, subgroup, company, sector, dig_ai, matched_keywords}

# 7. 요약 후
{..., summary, summary_ok, summary_fail_reason, summary_model}
```

## DB 스키마 (`news.db`)

**저장하는 것**: 반영 기사만 (`excluded=0`), 60일 보관
**저장하지 않는 것**: 제외 기사, 본문 전문

- 제외 기사는 메모리에서 HTML로 직접 전달 → DB 크기 6.4MB → 233KB (96% 감소)
- 본문을 저장하지 않으므로 `retry_summary.py`는 URL로 본문을 다시 긁는다

| 테이블 | 용도 |
|---|---|
| `articles` | 기사 (delivered=1이면 반영 완료) |
| `runs` | 회차별 요청 시각·통계 |

`cleanup()`은 60일 지난 기사와 남아 있는 제외 기사를 지운 뒤 `VACUUM`으로 파일을 축소한다.

## 병렬화

| 단계 | 병렬도 | 비고 |
|---|---|---|
| 네이버 수집 | 8스레드 | `requests.Session` 재사용 |
| 본문 추출 | 16스레드 | 타임아웃 5초 |
| 요약 | **1스레드** | Groq 레이트리밋 회피 — 병렬화하지 않는다 |

실행 시간 23분 → 6~10분으로 단축된 것이 이 병렬화 덕분이다.
