# 요약 & Groq 토큰 관리

---

## 요약 형식 — 개조식

```
- KB금융, 실전형 AI(인공지능) 인재 양성 프로그램 'KB AI Lab' 출범
- KB AI 아카데미 최고 과정 수료 직원 대상 프로젝트형 육성 프로그램
- 2주 심화 교육과 10주 실전 프로젝트로 구성
```

**규칙**

| 항목 | 내용 |
|---|---|
| 불릿 | `- ` 로 시작 (HTML에서는 `–` 로 표시) |
| 줄 수 | 2~3줄 |
| 종결 | 명사(구)로 끝냄 — 출범, 도입, 제공 예정, 추진 중 |
| 첫 줄 | "회사명, 핵심내용" |
| 협력사 | 첫 줄 끝에 `(w/회사명)` |
| 약어 | 처음 나올 때 `AI(인공지능)` 처럼 풀어씀 |
| 금지 | -입니다, -습니다, -한다, -했다, -이다, -됨, -함, -임 |

**줄마다 역할을 정하지 않는다.** 예전엔 "1줄 도입 / 2줄 변화 / 3줄 시사점"으로 지정했으나, 본문 사실만 고르게 담는 방식으로 바꿨다. 부수 효과로 프롬프트가 짧아지고 8b 모델도 충분히 해낼 수 있게 됐다.

---

## 검증 3단계

요약 결과는 세 관문을 통과해야 채택된다.

### 1. 형식 검증 (`_validate`)

- 2~3줄인가
- 각 줄이 `-` 로 시작하는가 (모델이 `*`를 쓰면 자동으로 `-`로 정규화)
- 각 줄이 15자 이상인가
- 서술형 어미로 끝나지 않는가 (`BAD_ENDINGS`)

### 2. 근거 검증 (`_grounded`) ★

**요약에 쓰인 단어가 원문에 실제로 등장하는지** 확인한다. 55% 미만이면 환각으로 판정하고 버린다.

```python
src = (title + " " + body).lower()
toks = re.findall(r"[가-힣]{2,}|[A-Za-z]{2,}", summary)
hits = sum(1 for t in toks if t.lower() in src)
return hits / len(toks) >= 0.55
```

**왜 필요한가**: 모델이 요약할 내용을 못 찾으면 **프롬프트의 예시를 그대로 복창한다.** 골드만삭스 기사와 JP모건 기사에 똑같은 "KB AI Lab" 요약이 나왔던 실제 사고가 있었다.

| 사례 | 겹침 | 판정 |
|---|---|---|
| 골드만삭스 기사 + KB AI Lab 예시 복창 | 0.00 | 차단 |
| 토스 기사 + 정상 요약 | 0.81 | 통과 |

예시 복창뿐 아니라 **없는 회사·없는 숫자를 지어내는 것도** 이 관문에서 걸린다.

### 3. 짧은 본문 사전 차단

```yaml
min_body_chars: 200
```
본문이 200자 미만이면 **Groq를 호출하지 않는다.** 토큰을 아끼면서 동시에 환각(예시 복창)의 근본 원인을 없앤다.

## 토큰 관리

### 현재 소비 구조 (건당 약 1,673토큰)

| 구성 | 토큰 | 비중 |
|---|---|---|
| 본문 1,100자 | ~1,155 | **69%** |
| 프롬프트 | 279 | 17% |
| 출력 | 180 | 11% |
| 제목 | 34 | 2% |

**본문이 압도적이다.** `body_chars`가 하루 처리량을 결정한다.

### 하루 처리량

Groq 무료 티어는 **모델별 일일 토큰 한도(TPD) 10만**. 한국시간 **오전 9시**(UTC 자정)에 리셋된다.

| | 건당 토큰 | 70b | 70b+8b |
|---|---|---|---|
| 최초 | 3,275 | 30건 | 61건 |
| 1차 개선 | 2,088 | 47건 | 95건 |
| **현재** | **1,673** | **59건** | **119건** |

하루 신규 기사가 20~30건이므로 **70b만으로 두 배 여유**가 있다.

### 절감 조치 4가지

**① 본문 정제 후 문장 단위 절단**

`compact_body()`가 상용구를 걷어낸 뒤 문장 경계로 자른다.

제거 대상: `[OO뉴스 김OO 기자]` 프리픽스, 이메일, `사진=OO` 캡션, 저작권 문구, 관련기사 목록
→ 실측 **37%가 이런 군더더기**였다.

문장 중간에서 자르면 모델이 끊긴 문장을 해석하느라 품질도 떨어지고 토큰도 낭비된다.

**② 프롬프트 압축** — 529 → 279토큰

**③ `temperature: 0`** — 결정적 출력 → 형식 오류가 줄어 재시도 자체가 감소

**④ 재시도 낭비 차단** ★

형식 검증 실패 시 **1회만 재시도**한다 (이전엔 3회 + 8b로 3회 = 최대 6회).

> 형식 오류는 같은 프롬프트로 다시 불러도 대개 같은 결과다. 이전 방식은 실패 1건에 **10,038토큰**(정상 6건치)을 태웠다. 실패 8건이면 그것만으로 하루 쿼터의 절반이 증발했다.

## 모델 폴백 체인

```
llama-3.3-70b-versatile  (주 모델)
        ↓ 일일 한도(TPD) 소진 시
llama-3.1-8b-instant     (대체 모델 — 쿼터 별개)
```

- **일일 한도(TPD)** 소진 → 재시도 없이 **즉시** 8b로 전환
- **분당 한도(TPM)** → 지수 백오프로 잠시 대기 후 재시도

두 모델의 쿼터가 별개라 합산 20만 토큰을 쓸 수 있다.

**8b 품질**: 새 프롬프트 기준 요약 성공률 94%. 단순 사실 요약이라 70b와 차이가 크지 않다. 예전에 "3번째 줄 시사점 추론"을 요구할 때는 격차가 있었으나, 그 요구를 없앤 뒤로는 실용상 문제없다.

## 사용량 확인

실행 완료 화면에 Groq가 실제 리턴한 수치가 표시된다 (추정 아님).

```
→ 토큰 사용 70b 42,000/100,000 (42%) · 8b 3,200/100,000 (3%)
```

| 표시 | 판단 |
|---|---|
| 40% 이하 | 여유 |
| 70~90% | 오늘은 한 번 더 정도 |
| 90% 이상 | 그만. 내일 오전 9시 이후 |

## 요약 실패 대응

### 실패 사유별

| 사유 | 의미 | 대응 |
|---|---|---|
| `레이트리밋(일일한도:모델명)` | 쿼터 소진 | 내일 오전 9시 이후 재실행 또는 `retry_summary.py` |
| `환각(원문 근거 없음)` | 예시 복창·지어냄 | 본문 추출 문제일 가능성 — URL 확인 |
| `본문 부족(N자)` | 본문 200자 미만 | 매체 페이지 구조 문제 |
| `형식 검증 실패` | 개조식 불충족 | 드묾. 반복되면 프롬프트 검토 |

### 실패분 재시도

```bash
python retry_summary.py         # 전체
python retry_summary.py 30      # 30건만 (쿼터 아껴 쓰기)
```

재수집 없이 DB의 실패 기사만 골라 본문을 다시 긁고 요약한다. 성공하면 HTML도 자동 갱신. 레이트리밋이 연속 3회 나면 쿼터 소진으로 판단해 자동 중단한다.

> **왜 별도 스크립트가 필요한가**: 요약에 실패해도 기사는 `delivered=1`로 저장된다. 다음 실행에서 다시 수집되면 "기열람"으로 제외되므로 재시도 기회가 없다. 요약 캐시도 `summary_ok=1`인 것만 재사용하므로 실패분은 캐시되지 않는다.

## 절대 규칙

**`batch: false` 를 `true`로 바꾸지 말 것.**

기사 여러 건을 한 프롬프트에 묶어 요약하면 토큰은 아끼지만, 모델이 일부 기사를 조용히 누락시킨다. 이것이 초기 기사 유실의 원인으로 진단됐다. **기사 1건 = 호출 1회**를 지킨다.
